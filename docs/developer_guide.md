# Python上位机开发指南

本文档面向后续开发人员，目标是让新增功能在不破坏现有 UDP/TCP/串口/文件能力的前提下，按统一架构稳定迭代。

配套扩展文档：

- 热力图/机器学习模块开发指南：[docs/heatmap_ml_module_guide.md](docs/heatmap_ml_module_guide.md)
- C/C++开发者进化手册（项目实战版）：[docs/cpp_to_python_evolution_manual.md](docs/cpp_to_python_evolution_manual.md)

## 1. 项目定位与目标

本项目是一个实时数据采集与可视化上位机，核心目标：

- 多数据源统一接入：UDP、TCP、串口（含 USB CDC/串口转TTL）、文件回放。
- 多协议兼容：文本协议、Justfloat（带时间戳/无时间戳）、Rawdata。
- 实时性能稳定：高频接收下 UI 可响应、状态机可收敛、日志可控。
- 可扩展：新增数据源/协议/分析功能时尽量不影响现有链路。

## 2. 当前能力总览

- 数据源：UDP、TCP、串口、文件（.log/.bin）。
- 协议：文本协议、Justfloat、Rawdata。
- 双向通信：串口/UDP/TCP 支持发送，文件源只读不发送。
- 文件源：支持实时尾随读取（tail），文件追加后可继续解析。
- 可视化：多通道时域显示，FFT 分析。
- 状态管理：连接状态 + 数据接收状态机（含格式错误态）。
- 回归测试：`tests/test_regression_layering.py` 持续覆盖核心分层行为。

## 3. 架构分层（必须遵守）

### L1 传输层（Transport）

位置：`src/data_sources/*_source.py` 的 socket/serial/file IO 部分

职责：

- 读取原始字节流。
- 发送原始字节流（若该源支持发送）。

禁止：

- 直接操作 UI。
- 更新业务状态机。
- 维护通道命名策略。

### L2 协议层（Parser）

位置：`src/data_sources/*_source.py` 中的 `_parse_*` 方法

职责：

- 将原始字节解析为元组/中间结构。
- 对格式错误返回可识别标识（如空元组、`FORMAT_ERROR`）。

禁止：

- 写入 UI 控件。
- 做跨源业务判断。

### L3 领域层（Domain/Manager）

位置：`src/data_sources/manager.py`

职责：

- 统一帧输出：`read_frame()`。
- 校验头检查、格式错误聚合。
- 通道管理、通道重命名映射。
- 缓冲与 CSV 保存触发。
- 统一发送入口：`send_data()`。

禁止：

- 直接依赖 UI 控件。

### L4 应用编排层（Application）

位置：`src/main.py`

职责：

- 连接/断开编排。
- 线程、队列、节流。
- 状态机事件分发。
- 从 UI 读取配置并创建数据源。

禁止：

- 重新实现协议解析。
- 在 UI 层复制一份通道命名真相。

### L5 表现层（UI）

位置：`src/main.py`、`src/visualization/waveform_widget.py`

职责：

- 可视化与用户交互。
- 将用户操作转成领域命令（如重命名、调色）。

禁止：

- 直接维护业务真相源。

## 4. 单一真相源（SSOT）

以下状态必须单点维护：

- 当前数据源：`DataSourceManager.current_source`
- 通道映射：`DataSourceManager.channel_name_mapping`
- 通道列表：`DataSourceManager.channels`
- 保存状态：`DataSourceManager.data_saver`

任何新功能都不能在 UI 再维护一套等价状态并双向同步。

## 5. 核心数据流

1. 数据源 `read_data()` 读取并解析。
2. Manager `read_frame()` 统一成帧：`header/timestamp/channels/meta`。
3. 接收线程将帧入队。
4. 主线程消费队列并更新状态机与波形。
5. 若开启保存，同步写 CSV。

发送链路：

1. UI 发送输入。
2. `DataSourceManager.send_data()` 路由到当前数据源。
3. 源实现 `send_data()` 完成协议通道发送。

## 6. 目录职责约定

- `src/main.py`：应用编排、状态机、UI整合。
- `src/data_sources/base.py`：数据源契约。
- `src/data_sources/manager.py`：统一帧与领域规则。
- `src/data_sources/*_source.py`：具体数据源实现。
- `src/visualization/waveform_widget.py`：波形与分析显示。
- `tools/*.py`：联调脚本（发送器、数据生成器）。
- `tests/test_regression_layering.py`：关键分层回归。

## 7. 新增功能开发方法

### 7.1 新增数据源（推荐流程）

1. 在 `base.py` 契约下实现新 Source：`connect/read_data/disconnect`。
2. 协议解析逻辑只放在 source 内。
3. 在 `manager.py` 增加工厂函数。
4. 在 `main.py` 的 `_build_data_source_from_ui` 增加配置分支。
5. 添加/更新回归测试。
6. 更新 `README.md` 与本指南。

### 7.2 新增协议（在已有源内扩展）

1. 在具体 source 新增解析分支。
2. 明确格式错误返回策略（空元组或 `FORMAT_ERROR`）。
3. 检查 `manager.read_frame()` 的格式敏感处理是否匹配。
4. 补充协议专用测试数据与回归断言。

### 7.3 新增 UI 功能

1. UI 只负责输入和展示。
2. 业务规则落到 Manager 或编排层私有方法。
3. 避免在高频路径加逐包日志。

## 8. 调试指南（Debug）

### 8.1 推荐调试顺序

1. 先看连接状态是否正确切换。
2. 再看 `read_frame()` 是否有有效帧输出。
3. 再看队列是否堆积。
4. 最后看 UI 渲染与按钮状态。

### 8.2 常见问题定位

#### A. 连接成功但无曲线

- 校验头不匹配。
- 协议选择错误（文本/Justfloat/Rawdata）。
- 文件模式下选错格式（`.log` 用文本协议，`.bin` 用 Justfloat）。

#### B. 状态灯和状态文字不一致

- 检查状态机事件是否进入预期分支。
- 检查接收态兜底同步逻辑是否触发。

#### C. 文件模式不更新

- 确认文件源是尾随模式。
- 确认写入端有 `flush`（或 close）动作。
- 文本协议确认每帧有换行符。

#### D. 高频下卡顿

- 关闭原始数据显示。
- 降低日志级别。
- 关注队列长度和丢包统计。

#### E. 点阵压力UDP有发送但上位机无数据

- 优先检查 UDP 接收缓冲是否足够承载整帧文本报文（16x16/32x32 点阵帧通常大于 1KB）。
- 若报文被截断，解析会失败并表现为“无数据/格式错误”。
- 对点阵文本协议建议使用大包接收缓冲（本项目默认已提升），并关注状态栏丢包指标。

### 8.3 调试日志规范

- 高频路径禁止无开关 `print`。
- 必须可通过开关控制调试输出。
- 异常日志保留，正常路径最小化。

## 9. 测试与验证规则

每次改动至少执行：

```bash
python -m unittest tests/test_regression_layering.py
```

若改动了协议或数据源，需重点验证：

- 切源后状态清理是否正确。
- Justfloat 重命名链是否收敛。
- `read_frame()` 与 `read_data()` 兼容。
- 文件实时尾随读取是否持续生效。
- 发送路由是否按当前数据源生效。

## 10. 变更前检查清单

- 是否破坏了分层边界？
- 是否新增了重复状态真相源？
- 是否影响其他数据源默认行为？
- 是否更新了回归测试？
- 是否同步更新用户文档与开发文档？

## 11. 提交与评审建议

- 提交信息要包含：影响层、风险点、验证结果。
- 单个提交尽量只做一类改动（例如只改数据源层或只改UI层）。
- 评审优先级：行为回归风险 > 代码风格。

## 11.1 Bug分析记录规范（强制执行）

每次修复 Bug 必须同步更新 `BUGLIST.md`，记录不可省略。

最低记录模板：

1. 现象：用户可观察到的异常表现。
2. 复现条件：数据源、协议、参数、触发步骤。
3. 分析过程：排查路径与关键证据。
4. 根因结论：明确单点或多点根因。
5. 修复方案：改动点与分层归属。
6. 验证结果：测试项、测试命令、通过结论。
7. 影响范围：是否影响 UDP/TCP/串口/文件其他链路。

评审门禁：

- 未更新 `BUGLIST.md` 的 Bug 修复提交，不允许合并。
- 涉及协议、时间戳、队列、状态机的修复，必须补充至少一条自动化回归用例。

## 12. 联调脚本建议用法

- UDP发送：`tools/udp_sender.py`
- TCP发送：`tools/tcp_sender.py`
- 文件生成：`tools/generate_test_files.py`
- 动脉训练：`tools/train_arterial_model.py --model-type rf|logreg|svm|gbdt`
- 训练脚本必须支持从项目根目录直接执行（`python tools/train_arterial_model.py`），不得依赖外部 `PYTHONPATH`；如使用 `src` 布局，需在脚本入口显式注入项目 `src` 路径。

机器学习中心（UI）约定：

- 机器学习相关控件统一放在“机器学习中心”，包含“在线推理配置”和“离线训练配置”。
- 在线推理必须支持模型类型下拉（自动/规则/随机森林/逻辑回归/SVM/梯度提升树）。
- 离线训练触发脚本后，训练完成应自动回填模型路径并应用推理配置。
- 离线训练回调必须使用 Qt 信号/槽跨线程回主线程，禁止在 Python 普通线程中依赖 `QTimer.singleShot` 直接回调 UI。
- 离线训练必须设置超时保护（建议 `subprocess.run(timeout=300)`）并提供“已运行耗时”可视反馈，避免用户误判界面卡死。
- 指标导出配置应采用紧凑交互（下拉多选）以减少面板高度占用，并支持手动指定导出 CSV 路径。
- 离线训练应暴露可配置参数（至少 `test_size`、`seed`），并允许通过“高级参数”透传模型超参数到训练脚本。
- 指标图表导出（PNG）完成后，UI必须保留“CSV/PNG导出结果路径”可见，不得在停止保存流程中被“未启用”文本覆盖。
- 当PNG未生成时，必须给出可理解原因（如无样本、无数值指标或保存失败），避免用户误判功能失效。
- 外部模型若使用带特征名数据训练，在线推理必须使用带列名的 DataFrame 输入（列名与顺序匹配 `feature_order`），避免 sklearn 特征名告警与潜在特征错位。
- 指标导出配置在“保存进行中”必须支持动态生效（启用即启动导出，停用即收尾导出并给出结果提示）。
- PNG导出依赖CSV数据，交互上应自动联动（勾选PNG时自动启用CSV导出）。
- 指标导出文件路径应在运行时解析为绝对路径（相对路径按项目根目录解释），避免不同启动目录导致“文件找不到”。
- 指标导出链路必须具备生命周期调试日志（start/append/stop/chart），便于现场定位“未生成”与“生成路径不一致”问题。
- 指标导出会话不得强依赖“开始保存”按钮；只要处于连接接收状态且勾选导出，就应自动启动导出。
- 断开连接时必须收尾导出会话，保证CSV/PNG在连接结束瞬间可见。
- 导出路径在会话运行中被修改时，必须即时生效（重启导出会话或刷新标签），避免“显示路径与实际路径不一致”。
- 当无有效数值趋势时，PNG导出应生成占位图并写明原因，避免用户误认为导出失败。
- 勾选“保存时同步导出趋势图(PNG)”时，UI必须立即提示导出目标路径与导出时机（停止保存/断开连接），并在标签区实时展示CSV/PNG目标路径。
- 在已连接状态下勾选PNG导出时，系统应即时确保指标导出会话已启动，避免出现“勾选成功但最终无CSV/PNG产物”。
- 仅启用CSV导出时也必须提供等价的即时反馈（目标路径+状态提示）；不得出现“仅勾选CSV无任何反馈、勾选PNG才有反馈”的交互不一致。
- 默认日志策略应以业务关键日志为主；FSM/UI_DEBUG 默认关闭，仅保留 Export 相关日志输出（可通过环境变量显式开启FSM调试）。
- 高吞吐跨机联调场景下，`update_data` 主循环必须设置单轮处理时间预算，并对压力热图渲染进行限频，防止主线程长时间占用导致界面卡顿。
- 当处理队列出现明显积压时，系统应切换“低延迟优先”策略：动态提高单轮消费配额，并在高水位下丢弃最旧待处理帧（追帧），避免出现“发送结束后界面仍回放多秒”的滞后体验。
- 在高吞吐场景中，动脉分析管线应采用“按轮取最新帧+时间间隔降频”策略，避免每帧都做分析导致处理速率低于接收速率；波形接收可继续全量，分析结果优先实时性。
- 离线训练必须在启动线程前完成前置校验：数据集存在、文件类型为CSV、输出目录可写；校验失败应直接阻断并提示。
- 打包版训练应优先走内置训练服务（同进程调用），避免 `sys.executable` 自启动上位机窗口。
- 若内置训练能力缺失，才回退外部解释器方案，并给出明确配置提示。

文件实时联调建议：

1. 上位机选“文件”数据源并加载目标文件。
2. 用 `generate_test_files.py --live` 持续写入同一文件。
3. 观察波形与状态是否持续更新。
