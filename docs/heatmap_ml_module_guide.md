# 热力图/机器学习模块开发指南

本文档用于指导在现有上位机架构中新增“热力图分析 + 机器学习分析”能力，目标是在不破坏 UDP/TCP/串口/文件链路稳定性的前提下，分阶段交付可落地功能。

配套文档：

- 任务拆解：[docs/arterial_pressure_task_breakdown.md](docs/arterial_pressure_task_breakdown.md)
- 开发手册：[docs/arterial_pressure_dev_manual.md](docs/arterial_pressure_dev_manual.md)

## 1. 目标与范围

本次扩展聚焦“数据处理模块增强”，包含两类能力：

- 热力图能力：将多通道时间序列映射为二维强度图，用于观察通道-时间、频段-时间的能量分布。
- 机器学习能力：对窗口数据进行特征提取、推理与告警（如状态分类、异常检测、趋势评分）。

热力图维度说明（先统一认知）：

- 本指南默认的“热力图”是二维强度图（2D）。
- 对于压力阵列场景，优先采用“空间二维平面”：X轴为列坐标，Y轴为行坐标，颜色表示该点压力值。
- 如果需要三维可视化，可在后续阶段增加“3D曲面/体渲染”视图，但底层数据仍建议复用同一二维压力矩阵（避免破坏实时链路）。

不在本阶段范围：

- 替换现有数据源协议解析逻辑。
- 重写现有波形组件。
- 强制引入重型训练框架（训练可离线，推理先在线）。

## 2. 现有架构约束（必须遵守）

项目当前已形成分层：

- L1 传输层：DataSource 读取原始字节。
- L2 协议层：各 source 完成 parse。
- L3 领域层：DataSourceManager 输出统一帧 read_frame。
- L4 应用编排层：接收线程 + 队列 + 状态机 + UI更新。
- L5 表现层：波形/原始数据/交互。

新增分析能力时必须遵守：

- 不修改 L1/L2 的协议行为与接口语义。
- 不在 UI 层维护第二套业务真相源。
- 分析模块只消费统一帧，不反向依赖具体 source 类型。

## 3. 建议扩展架构

建议引入“分析子系统”作为 L3 与 L4 之间的旁路模块（可视作 L3.5）：

- 输入：统一帧 `{header, timestamp, channels, meta}`。
- 处理：窗口缓存、特征提取、热力图栅格化、模型推理。
- 输出：`AnalysisResult`（热力图结果、特征结果、模型结果、健康状态）。

推荐目录（新增）：

```text
src/
  analytics/
    __init__.py
    contracts.py              # 输入/输出数据契约
    pipeline.py               # 分析主管线编排
    window_buffer.py          # 滑动窗口缓存
    heatmap/
      __init__.py
      builder.py              # 热力图构建器
      normalizer.py           # 强度归一化/色阶策略
    ml/
      __init__.py
      feature_extractor.py    # 特征提取
      model_runner.py         # 模型加载与推理
      postprocess.py          # 阈值判定、标签映射、平滑
```

可选目录（第二阶段再做）：

```text
src/core/
  analysis_flow_mixin.py      # 主窗口分析流程编排（可复用现有mixin风格）
```

## 4. 数据契约设计

### 4.1 输入契约

分析模块统一消费 read_frame 输出：

```python
{
    "header": str,
    "timestamp": float,   # ms
    "channels": {"ch1": 1.23, "ch2": 2.34},
    "meta": {"format_error": bool, "protocol": str}
}
```

处理规则：

- `meta.format_error=True` 时跳过分析，仅更新内部错误计数。
- `channels` 为空时不触发热力图/ML计算。
- `timestamp` 必须单调处理，出现回退时记录告警并按接收时序兜底。

### 4.2 输出契约

建议定义统一输出结构：

```python
AnalysisResult = {
    "timestamp": float,
    "heatmap": {
        "matrix": np.ndarray | None,
        "x_axis": list,
        "y_axis": list,
        "meta": dict,
    },
    "features": {
        "values": dict,
        "window": dict,
    },
    "prediction": {
        "label": str,
        "score": float,
        "topk": list,
        "anomaly": bool,
    },
    "health": {
        "dropped_frames": int,
        "latency_ms": float,
        "warnings": list,
    }
}
```

### 4.3 上位机接入协议建议

本项目已支持文本协议与 Justfloat 协议。为保证兼容，推荐继续沿用现有协议，不新增专用“热力图协议”。

推荐方案A：文本协议（联调友好）

- 每帧结构：`HEADER,timestamp,ch1=v1,ch2=v2,...`
- 示例：`DATA,1710000000.123,ax=0.12,ay=-0.05,az=9.81,temp=36.5`
- 适用场景：快速联调、日志可读、跨语言调试。

推荐方案B：Justfloat（高吞吐）

- 负载：`float[] + tail(00 00 80 7f)`。
- 无时间戳模式：时间由上位机按 delta_t 推算。
- 带时间戳模式：帧内最后一个 float 为时间戳。
- 适用场景：高采样率、低文本解析开销。

协议侧约束：

- header 校验由 DataSourceManager 执行，失败帧标记 `format_error`。
- 通道名以 Manager 最终映射为准，分析模块只消费映射后的 channels。
- 机器学习输入建议固定通道集合，缺失值补零并记录 mask。

### 4.4 解析与映射规则（用于热力图与ML）

1. 数据源层解析原始报文，产出基础帧。
2. Manager 输出统一帧：`header/timestamp/channels/meta`。
3. 分析管线只接收 `meta.format_error=False` 且 `channels` 非空的帧。
4. WindowBuffer 组窗：
  - 长度为 `analysis_window_size`。
  - 步长为 `analysis_stride`。
5. 热力图构建：
  - 通道热力图：矩阵形状 `C x T`（通道数 x 时间点）。
  - 频谱热力图：矩阵形状 `F x T`（频率Bin x 时间窗）。
6. 机器学习输入：
  - 路径1：直接用标准化后的窗口序列。
  - 路径2：先提取特征再推理（均值、方差、RMS、峰值、带宽能量等）。
7. 输出 AnalysisResult 给 UI：
  - heatmap.matrix 用于绘图。
  - prediction 用于标签、分数、告警显示。

### 4.5 二维平面压力分布（本项目优先实现）

目标：

- 在上位机显示一个固定尺寸的二维平面（例如 16x16、32x32）。
- 平面上每个点对应一个压力传感点。
- 点颜色由压力值映射得到，形成实时压力分布图。

推荐点阵定义：

- `grid_width`: 列数（X方向）。
- `grid_height`: 行数（Y方向）。
- `point_count = grid_width * grid_height`。
- 点位索引采用行优先：`idx = row * grid_width + col`。

颜色映射建议：

- 统一最小值 `vmin` 与最大值 `vmax`，防止画面闪烁。
- 压力值先归一化：`norm = clip((p - vmin) / (vmax - vmin), 0, 1)`。
- 再映射到 colormap（如 Turbo/Viridis）。

异常值处理：

- 缺失点填充为 NaN 并在画面上用固定颜色表示（如灰色）。
- 越界值先裁剪再绘图。
- 单帧点数不足时整帧丢弃并记录告警计数。

### 4.6 下位机到上位机的推荐报文

方案A（强推荐，最少改动）：文本协议点位命名

- 每帧格式：`DATA,timestamp,p_row_col=value,...`
- 示例（2x3）：
  `DATA,1710000000.123,p_0_0=12.1,p_0_1=13.5,p_0_2=15.0,p_1_0=10.2,p_1_1=9.8,p_1_2=11.4`

优点：

- 与现有文本协议兼容，解析链路改动小。
- 直接由 key 提取 row/col，映射关系清晰。

方案B（高吞吐）：Justfloat固定点阵

- 每帧携带 `point_count` 个 float（可选附带时间戳float）。
- 上位机通过已配置的 `grid_width/grid_height` 重建二维矩阵。
- 映射：第 k 个值对应 `(row = k // grid_width, col = k % grid_width)`。

优点：

- 开销低，适合高刷新率点阵压力数据。

### 4.7 解析到绘图再到ML的压力数据链路

1. 下位机发送一帧点阵压力数据（文本或Justfloat）。
2. DataSource 解析后输出统一帧 channels。
3. PressureGridAdapter 将 channels 转为 `pressure_matrix[H][W]`。
4. UI 热力图组件直接渲染该矩阵（每个网格点一色）。
5. 同一矩阵进入 ML：
  - 路径1：直接输入模型（图像型输入）。
  - 路径2：提取特征后输入模型（均值、重心、接触面积、高压区域比例、区域能量等）。
6. 预测结果返回 UI，显示类别、置信度、异常状态。

推荐新增适配层：

- `analytics/heatmap/pressure_grid_adapter.py`
- 职责：仅做 channels 到二维矩阵的重排与校验，不做模型推理。

## 5. 主链路集成点

### 5.1 接收线程

现状：`DataReceiveThread` 只负责 `read_frame -> queue.put`。

建议：

- 保持线程职责不变，不在接收线程做重计算。
- 分析计算放到主线程定时批处理后的“分析线程/分析定时器”。

### 5.2 主窗口数据处理

现状：`MainWindow.update_data` 已有批处理。

建议插入点：

- 在 `_update_waveform_from_packet(data_dict)` 之后，调用 `analysis_pipeline.submit_frame(data_dict)`。
- 通过节流策略控制分析频率（例如每 20ms 或每 N 帧触发一次）。

### 5.3 可视化层

现状：`WaveformWidget` 已有“时域/频域”Tab。

建议：

- 新增“热力图”Tab（可在现有组件内增 tab，或拆分独立 widget）。
- 机器学习结果先以“状态条 + 结果标签 + 分数”呈现，不先做复杂可视化。
- UI 只展示 `AnalysisResult`，不做算法逻辑。

### 5.4 从报文到绘图再到ML的最小闭环

建议按下列最小闭环实现并验收：

1. 发送端按文本协议或 Justfloat 持续发送点阵压力数据（每帧包含全部点位）。
2. 上位机现有链路完成解析并显示时域曲线。
3. 分析管线每 N 帧生成一次二维 `pressure_matrix`，在热力图Tab刷新。
4. 同一矩阵并行进入 `feature_extractor -> model_runner`。
5. UI 同步显示 `label/score/anomaly`，并记录推理耗时。

最小验收标准：

- 数据不断流时，热力图持续更新且不明显卡顿。
- 断流或格式错帧时，热力图与ML结果安全降级，不影响基础波形。
- 切换 UDP/TCP/串口/文件时，分析缓存可重置且不串数据。

## 6. 性能与线程模型

### 6.1 性能预算

建议先定义预算，避免功能上线后拖垮实时链路：

- 接收线程：不引入额外阻塞，单帧处理开销保持近似现状。
- 主线程：`update_data` 每轮批处理不新增重型计算。
- 分析线程：单窗口推理控制在可配置阈值内（例如 < 15ms）。
- UI 刷新：热力图刷新频率限制在 5-20Hz。

### 6.2 线程策略

推荐策略：

- 主线程负责消费数据队列与更新基础UI。
- 分析线程消费“分析输入队列”，输出结果到“分析结果队列”。
- 主线程定时读取分析结果队列并渲染。

注意事项：

- 不跨线程直接操作 Qt 控件。
- 队列满时采用“丢旧保新”策略。
- 模型加载失败不影响主接收链路，必须降级为“分析禁用”。

## 10. 风险与回滚策略

主要风险：

- 高频数据下分析线程积压导致结果滞后。
- 模型初始化阻塞 UI。
- 通道动态重命名导致特征维度漂移。

规避策略：

- 分析输入前执行通道对齐与缺失补零。
- 模型预热放在后台线程。
- 所有分析功能受开关保护，可运行时禁用。

回滚策略：

- 出现性能退化时，立即关闭 `analysis_enabled`。
- 保留既有波形/保存链路不受影响。

## 13. 机器学习落地手册（训练与使用）

本节给出可直接执行的机器学习方案，目标是先稳定上线，再逐步提升精度。

### 13.1 先确定任务类型

压力点阵最常见三类任务：

- 分类：识别状态类别（如空载、正常接触、偏载、异常）。
- 回归：预测连续量（如总压力、接触面积、中心位置偏移量）。
- 异常检测：未见过模式告警（无监督或一类学习）。

建议上线顺序：分类优先，异常检测次之，回归按业务价值补充。

### 13.2 数据采集与标注规范

最小采集单元建议：

- 一条样本 = 一个时间窗口（例如 1-2 秒）内的多帧压力矩阵序列。
- 每条样本包含：
  - matrix_seq: 形状 T x H x W。
  - label: 类别或回归目标。
  - meta: 数据源、设备编号、采样率、时间戳范围、工况标签。

标注建议：

- 先定义标签字典与边界条件，避免后期标签漂移。
- 同一工况至少覆盖多设备、多天、多操作者数据。
- 标注冲突样本单独入库，做人工复核集。

数据切分建议：

- 按采集会话或设备切分训练/验证/测试，避免同源泄漏。
- 不建议随机逐帧切分，否则离线指标会虚高。

### 13.3 特征与模型基线

基线A（最快落地）：传统机器学习

- 特征：
  - 全局统计：均值、方差、最大值、最小值、RMS、分位数。
  - 空间特征：压力重心(x,y)、高压区面积、连通域数量。
  - 时间特征：窗口内变化率、峰值持续时间、波动指数。
- 模型：随机森林、梯度提升树、线性模型。
- 适用：样本量较小、需要快速解释与上线。

基线B（中期提升）：轻量深度学习

- 输入：T x H x W 压力序列。
- 模型：小型 2D CNN（逐帧）或 3D CNN/ConvLSTM（时空联合）。
- 适用：样本量较大、模式更复杂。

建议先上基线A，稳定后再引入基线B。

### 13.4 训练流程

1. 数据清洗

- 处理缺失点、坏点、异常跳变。
- 统一单位与量程，保存归一化参数。

2. 样本构建

- 按窗口长度与步长生成样本。
- 过滤有效帧占比过低的样本。

3. 模型训练

- 先做交叉验证选超参，再在全训练集重训。
- 记录训练配置、特征版本、模型版本。

4. 模型评估

- 分类指标：F1、召回率、混淆矩阵。
- 回归指标：MAE、RMSE。
- 异常检测：AUROC、误报率、漏报率。

5. 模型导出

- 导出模型文件（如 joblib/onnx）。
- 同步导出 metadata（特征顺序、标签映射、归一化参数、训练集版本）。

### 13.5 上位机推理接入

推理流程：

1. PressureGridAdapter 产出当前帧二维矩阵。
2. WindowBuffer 聚合为窗口样本。
3. FeatureExtractor 生成特征向量或模型输入张量。
4. ModelRunner 执行推理并返回 score/topk。
5. Postprocess 做阈值、平滑、滞回，输出最终状态。

上线建议：

- 采用双阈值滞回，避免状态抖动。
- 对预测结果做时间平滑（滑动平均或投票）。
- 推理异常时只降级 ML，不影响热力图和基础接收链路。

### 13.6 线上迭代与再训练

建议建立闭环：

- 在线记录低置信度样本与误报样本。
- 周期性人工复标后回灌训练集。
- 每次再训练生成新模型版本并灰度发布。

模型版本管理建议：

- 版本号包含日期与数据集版本。
- 上位机显示当前模型版本与更新时间。
- 保留至少一个稳定回滚版本。

### 13.7 最小可用实施方案（MVP）

第一版可按以下最小集合上线：

- 输入：单帧或短窗口压力矩阵。
- 特征：全局统计 + 重心 + 高压面积。
- 模型：随机森林分类。
- 输出：label + score + anomaly。
- 防抖：3窗口投票。

该方案优点是：实现快、解释性强、上线风险低。

如果迭代中修复了Bug，必须同步更新 `BUGLIST.md`，并在开发文档中补充影响范围与验证命令。

## 14. 动脉压力健康评估专用方案

本节面向你的目标场景：下位机采集动脉压力信号，上位机以二维皮肤表面分布动画展示，并结合机器学习评估动脉健康状态。

注意：本模块输出为工程评估结果，不应直接替代临床诊断结论。

### 14.1 业务目标拆解

可观察目标：

- 波动次数：单位时间内脉搏波个数。
- 波动情况：单搏形态、上升沿/下降沿、峰值与谷值变化。
- 一致性：同一时刻空间分布是否稳定（跨点阵一致）。
- 重复性：连续多个心搏在形态与时序上是否可重复。

机器学习目标：

- 融合空间、频率、时间特征，输出健康评分与风险等级。
- 输出可解释指标，支持回看与趋势对比。

### 14.2 动画展示定义（二维皮肤表面）

显示对象：

- 每帧是一个 H x W 的压力矩阵，代表皮肤表面瞬时压力分布。
- 按时间连续播放，形成动脉压力传播动画。

渲染约束：

- 建议刷新率 15-30 FPS。
- 颜色映射固定量程或慢更新量程，避免闪烁。
- 提供时间轴回放与暂停，支持逐搏查看。

建议叠加层：

- 主脉路径高亮（可选）。
- 峰值点轨迹。
- 每搏边界线（R-R 类似概念，可称 pulse-to-pulse）。

### 14.3 采集与协议扩展建议

下位机每帧除点阵压力外，建议附加元信息：

- frame_id：递增帧号。
- sensor_rate_hz：采样率。
- device_time_ms：设备时间。
- quality_flag：采样质量位（接触不良、饱和、丢点）。

文本协议示例：

- `DATA,ts=1710000000.123,frame=1024,rate=100,q=0,p_0_0=12.1,p_0_1=13.5,...`

Justfloat模式建议：

- 固定头部元信息字段 + point_count 压力值 + 帧尾。
- 上位机配置 grid_width/grid_height 与字段顺序，确保稳定解析。

### 14.4 指标体系（先规则后模型）

1. 时间域指标

- pulse_count_per_min：波动次数（每分钟搏动数）。
- upstroke_time / downstroke_time：上升与回落时间。
- pulse_amplitude：搏动幅值。
- beat_interval_cv：搏间期变异系数。

2. 空间域指标

- pressure_centroid_trajectory：压力重心轨迹。
- spatial_uniformity_index：空间一致性指数。
- high_pressure_area_ratio：高压区域面积占比。
- bilateral_symmetry_score：左右对称性评分（如有左右结构定义）。

3. 频域指标

- dominant_frequency：主频。
- harmonic_ratio：谐波能量比。
- band_energy_ratio：低中高频带能量占比。

4. 重复性指标

- beat_template_corr：相邻心搏模板相关系数。
- cycle_repeatability_score：周期重复性评分。
- spatial_repeatability_score：空间分布重复性评分。

### 14.5 机器学习实施方案

阶段1：规则基线

- 先计算上述指标，给出可解释评分。
- 使用阈值与规则引擎生成初始风险等级。

阶段2：监督学习

- 输入：窗口级别特征向量（空间+频率+时间）。
- 任务：多分类（正常/可疑/异常）或回归（健康分）。
- 模型：随机森林或梯度提升树优先。

阶段3：时空模型

- 输入：T x H x W 压力序列。
- 模型：轻量 CNN/ConvLSTM 或 3D CNN。
- 目标：捕捉传播模式与节律异常。

### 14.6 训练数据与标注要求

样本定义：

- 一个样本至少包含多个完整心搏周期。
- 每个样本保存原始矩阵序列、导出特征、标签和质量标记。

标注建议：

- 标签分层：健康等级标签 + 具体异常类型标签。
- 标注来源应包含专家判读或标准设备对照结果。
- 低质量样本不直接用于主训练集，可用于鲁棒性训练子集。

防数据泄漏：

- 按受试者或采集会话分割训练/验证/测试。
- 同一受试者数据不得跨集合混用。

### 14.7 上位机在线评估流程

1. 接收实时点阵帧并生成二维压力矩阵。
2. 实时动画显示皮肤表面压力传播。
3. 在线分段得到单搏周期，计算规则指标。
4. 将窗口特征送入模型得到风险输出。
5. 输出：
  - 健康评分（0-100）。
  - 风险等级（低/中/高）。
  - 关键依据指标（便于解释）。
6. 保存评估日志，支持趋势回放与重复性比较。

### 14.8 一致性与重复性评估实现要点

一致性（同一时刻空间稳定）：

- 对相邻帧做空间相关系数或结构相似度。
- 统计短窗内方差，得到一致性分数。

重复性（不同心搏之间可重复）：

- 先做搏动对齐（按峰值或模板匹配）。
- 计算搏动间 DTW 距离或相关系数。
- 输出重复性分数与异常搏动占比。

### 14.9 最小可交付版本（面向当前项目）

第一版建议交付：

- 实时二维压力动画（固定点阵）。
- 自动波动计数与幅值曲线。
- 一致性/重复性两项基础评分。
- 轻量模型风险分级（低/中/高）。
- 评估结果导出（CSV/JSON）。

这样能在较小风险下先完成“可观察 + 可量化 + 可评估”的闭环。
