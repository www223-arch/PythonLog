## 2026-03-14 实时性性能问题排查记录

### 现象
- 用户侧体感明显慢于 VOFA。
- 在 `1ms/点` 发送条件下，状态栏接收点数增长慢，最初约 `180/s`。
- 优化后最高提升到约 `710-720/s`。

### 分析过程（按链路）
1. 先排查日志输出开销
- 发现高频路径存在大量 `print`，先做降噪（保留异常日志，移除每点日志）。

2. 检查线程与队列结构
- 引入后台接收线程后，确认存在队列积压与满队列丢包风险。
- 增加状态栏性能统计，实时观察：接收速率、处理速率、队列长度、丢包数。

3. 检查主线程处理与 UI 刷新
- 发现每点 UI 文本更新与频繁状态切换会消耗主线程。
- 改为批处理和限频刷新，减少每点 UI 成本。

4. 深挖串口协议解析
- 关键瓶颈：`Justfloat` 路径原先单次读取可能包含多帧，但只返回 1 帧。
- 导致吞吐被循环频率卡住。
- 改为“单次读取拆多帧入队，后续优先出队已解析帧”。

5. 增加物理链路判定指标
- 新增 `字节速率(B/s)` 与 `解析耗时(us/帧)`。
- 实测 `字节 11400-11600 B/s`，接近 `115200` 串口理论上限（约 `11520 B/s`）。

### 根因结论
- 主要瓶颈不是 UI，而是串口链路带宽上限 + 协议解析路径效率。
- 在 `115200` 下达到 `720/s` 已接近当前帧格式可达上限。
- `解析=0us/帧`（取整后）说明解析开销已不是主瓶颈。

### 已实施优化
1. 高频日志降载
- 移除数据循环中的每点日志，避免控制台 I/O 拖慢。

2. 主线程处理优化
- `update_data` 批处理、状态栏限频刷新。
- 自动保存默认不随连接自动开启，避免默认写盘影响吞吐。

3. 队列策略优化
- 满队列时“丢最旧保最新”，降低可见延迟。

4. 原始数据显示优化
- 原始数据显示改为可开关（默认关闭）。
- 改为线程安全队列 + 主线程批量刷新。

5. 串口接收解析优化
- `timeout=0` 非阻塞。
- `Justfloat` 支持单次读取多帧解析并缓存输出。
- 文本协议改为批量读取 + 行缓冲拆包，替代逐次 `readline`。

6. 热路径结构优化
- 通道去重由列表查找改为集合查找，降低高频 membership 开销。

7. 可观测性增强
- 状态栏新增：接收/s、处理/s、队列、丢包、字节B/s、解析us/帧。

### 结果
- 接收速率从约 `180/s` 提升到约 `710-720/s`。
- 在 `115200` 下，字节速率稳定在 `11400-11600 B/s`，符合物理上限预期。

### 后续建议
1. 若目标 > 1000 点/s
- 提升波特率（230400/460800/921600）。
- 或进一步减少每点字节数（优先二进制协议）。

2. 若继续追求极限实时

## 2026-03-16 文件CSV导入“只有几个点/长直线”排查记录

### 现象
- 导入静态 `.csv` 后，波形看起来只有少量点。
- 中间出现长直线，视觉上像“缺点”或“跳点”。

### 分析过程
1. 先核对 CSV 解析路径与时间戳语义
- 发现 DataSaver 导出的时间戳是毫秒。
- CSV 导入路径最初按“秒”处理，进入 Manager 后又统一乘以 1000，导致时间轴尺度异常。

2. 再排查线程与队列背压
- 静态文件回放速度远高于 UI 消费速度。
- 接收线程会瞬时灌入大量帧，`data_queue` 容量过小（原 300）时容易溢出。
- 满队列策略为“丢最旧保最新”，会表现为采样点稀疏、曲线跨大段直连。

### 根因结论
- 根因A：CSV 时间戳单位不一致（毫秒/秒转换错误），引发时间轴显示异常。
- 根因B：静态文件回放无节流 + 队列容量偏小，引发丢帧，波形被长线连接。

### 修复方案
1. 统一 CSV 时间戳单位
- 在文件源 CSV 解析处把 CSV 时间戳按毫秒读取，再换算成秒返回给 Manager。
- 保持 Manager 的统一语义（上游秒、下游毫秒）不变。

2. 降低静态文件回放丢帧概率
- 接收线程对文件源增加轻微节流（每帧约 0.5ms）。
- `data_queue` 容量从 300 提升到 5000。

3. 测试补齐
- 新增 CSV 时间戳尺度回归测试。
- 新增 CSV 多行全量读取测试（100 行必须读满）。

### 验证结果
- 回归测试通过（16 项）。
- 现场复测反馈：问题已解决。

### 其他格式是否存在同类问题（影响分析）
1. 文件源 `.log/.bin`
- 结论：存在同类风险（中高概率），机制与本次 CSV 一致。
- 原因：同属“静态文件回放”路径，生产速度可能远高于 UI 消费速度，队列满时会丢最旧帧，表现为点稀疏、长线连接。
- 当前缓解：本次在接收线程对文件源统一加轻微节流，并提升 `data_queue` 容量，因此 `.log/.bin/.csv` 已同步受益。
 
2. UDP/TCP/串口（实时链路）
- 结论：存在相似现象的可能（中低概率），但触发机制不同。
- 原因：若输入频率持续超过链路+解析+UI总吞吐，队列同样可能溢出并触发丢帧策略，可能看到“跨点连线”。
- 差异：实时链路通常受物理链路/发送速率约束，不像静态文件可瞬时灌入大量历史数据。

3. 时间戳单位类问题
- 结论：当前仅在 CSV 导入链路确认过该风险并已修复。
- 原因：`.log` 文本协议与 `.bin justfloat` 时间语义与 CSV 导出格式不同，不共享 CSV 的“毫秒列再换算”路径。
- 建议：后续若新增文件协议，必须在协议定义中显式声明时间戳单位，并补对应回归测试。

4. 后续监控建议
- 观察状态栏 `队列` 与 `丢包` 指标；若长时间增长，优先判定为背压问题。
- 对文件回放场景，必要时增加“按时间戳节奏回放”模式，进一步降低视觉稀疏感。

### 经验沉淀
- 文件回放链路与实时链路在“生产/消费速率比”上差异明显，必须单独考虑背压策略。
- 任何涉及时间戳的协议扩展都必须明确单位并在测试中锁定。

## 2026-03-16 UDP发送5000Hz时上位机卡顿分析

### 现象
- 将 `tools/udp_sender.py` 的发送频率提升到 `5000Hz` 后，上位机出现明显卡顿。
- 体感表现为：UI响应变慢、波形刷新不流畅，可能伴随丢包增加。

### 复现条件
- 数据源：UDP 文本协议。
- 发送速率：`5000` 点/秒（3通道时约 `15000` 通道值/秒）。
- 接收端：默认实时绘图与状态刷新逻辑开启。

### 分析过程
1. 发包路径检查
- `udp_sender` 正弦发送循环按 `sample_rate` 控制 `sleep` 间隔，速率提升会直接放大接收与解析压力。
- 发送器参数中存在 `--rate`，但 `send_sine_wave` 内历史实现使用固定 `sample_rate`（工具侧存在速率控制一致性风险）。

2. 协议解析路径检查（UDP Source）
- 每个 UDP 包会走文本解析：`split(',') + float()`。
- 解析后 `get_channel_names()` 仍基于 `last_raw_text` 再次 `split(',')` 提取通道名，存在重复字符串解析开销。

3. 应用层消费能力检查（Main/UI）
- 主线程 `update_data` 每 5ms 触发，每轮上限 `500` 帧，理论上限足够高。
- 但高频下瓶颈更多体现在：Python 文本解析成本 + 高频队列搬运 + Qt/pyqtgraph 重绘开销叠加。
- 波形组件以 `20ms` 刷新一次显示，输入过高时主线程在“处理数据+渲染”之间竞争，产生卡顿体感。

### 根因结论
- 根因A：UDP 文本协议在 `5000Hz` 下解析成本显著上升（字符串切分+浮点转换）。
- 根因B：通道名提取存在重复解析，放大 CPU 消耗。
- 根因C：高频数据处理与绘图刷新叠加，主线程时间片不足，导致 UI 卡顿。

### 影响范围
- UDP 文本协议：高风险（本次直接命中）。
- TCP 文本协议：中高风险（解析路径类似，速率足够高时可复现同类问题）。
- 串口文本协议：中风险（受物理带宽限制，较难到 5000Hz，但机制相同）。
- 二进制协议（如 justfloat）：相对低风险（解析开销更低），但极限速率下仍可能受 UI 渲染瓶颈影响。

### 优化建议（按优先级）
1. 协议优化优先
- 高速场景优先使用二进制协议（justfloat）替代文本协议。

2. 解析路径降耗
- UDPSource 在 `_parse_text_data` 阶段一次性缓存通道名，`get_channel_names()` 直接返回缓存，避免重复 `split`。

3. 可视化限载
- 高速模式下提高绘图刷新间隔（如 20ms -> 30/40ms）或降低每轮绘图复杂度。

4. 发送工具一致性
- 统一 `udp_sender` 的 `--rate` 参数与实际发送速率实现，避免测试速率与预期不一致。

### 验证建议
- 观察状态栏 `接收/s、处理/s、队列、丢包` 四项联动。
- 若 `队列` 持续上升且 `丢包` 增加，优先判定为吞吐超过当前实时渲染能力。

## 2026-03-16 连接时左侧控制面板宽度跳变与布局体验优化

### 现象
- 点击连接/切换协议时，左侧控制面板宽度偶发“跳变”，视觉卡顿明显。
- 用户希望：
	1. 面板尺寸可调且可锁定；
	2. 每个大块支持拖拽分离；
	3. 支持一键恢复默认布局与尺寸。

### 分析过程
1. 原布局结构
- 主界面使用 `QHBoxLayout` 固定比例（左1:右3）+ 右侧 `QSplitter`。
- 左侧控制面板内多个组在连接过程中会显隐与内容变化，布局管理器在比例约束下会反复重算宽度。

2. 跳变触发机制
- 控制面板内容高度/最小宽度变化（例如不同协议组显隐）会触发主布局重新分配宽度。
- 由于不是用户显式拖拽，而是布局重算，体感为“卡一下、宽度突然变”。

3. 方案比较
- 仅加 `QSplitter`：可调但不能自然支持“分离面板”。
- 改为 `QDockWidget`：天然支持拖拽重排、浮动分离、状态保存/恢复，更符合需求。

### 根因结论
- 根因：左侧面板处于比例布局中，且内部分组动态显隐导致宽度约束变化，触发非用户预期的重布局跳变。

### 修复方案
1. 布局架构改造
- 将三大块改为 Dock 面板：控制面板、波形区、原始数据与发送区。
- 支持拖拽重排、独立浮动分离。

2. 尺寸锁定
- 新增“锁定布局”动作：按当前尺寸冻结各 Dock（禁止拖拽分离，并固定当前宽/高）。

3. 一键复原
- 保存默认 `saveGeometry/saveState`，新增“一键复原布局”动作，恢复初始布局与尺寸。

4. UI观感优化
- 新增布局工具栏与 Dock 标题样式（渐变与圆角按钮），避免“纯文本+生硬按钮”观感。

### 验证结果
- 代码静态检查通过（`src/main.py` 无语法错误）。
- 交互层面可完成：拖拽分离、尺寸锁定、一键恢复。

### 影响范围
- 仅涉及 UI 布局与交互层，不改变数据源解析、线程队列与协议行为。
